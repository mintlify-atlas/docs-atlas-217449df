FRRouting (FRR) is a free, open-source routing software suite that implements and manages various IPv4 and IPv6 routing protocols. It runs on nearly all distributions of Linux and BSD as well as Solaris, and supports all modern CPU architectures. FRR is a fork of [Quagga](http://www.quagga.net/) and is distributed under the GNU General Public License.

A system with FRR installed acts as a dedicated router. FRR exchanges routing information with other routers using standard routing protocols, then uses that information to update the kernel routing table so that packets reach the right destination. You can dynamically change the configuration and view routing table information through the FRR terminal interface.

<Note>
  In the SONiC context, FRR serves as the routing protocol stack that interacts with the SONiC switch abstraction layer. SONiC uses FRR's daemons to handle BGP peering, OSPF convergence, and other routing functions, with route updates propagated through the SONiC database to the underlying ASIC.
</Note>

## System architecture

Unlike traditional routing software that bundles all protocol logic into a single monolithic process, FRR takes a multi-daemon approach. Each major routing protocol runs as its own independent daemon. A central broker daemon â€” **Zebra** â€” sits between the protocol daemons and the operating system kernel.

```
+----+  +----+  +-----+  +----+  +----+  +----+  +-----+
|bgpd|  |ripd|  |ospfd|  |ldpd|  |pbrd|  |pimd|  |.....|
+----+  +----+  +-----+  +----+  +----+  +----+  +-----+
     |       |        |       |       |       |        |
+----v-------v--------v-------v-------v-------v--------v--+
|                                                         |
|                          Zebra                          |
|                                                         |
+---------------------------------------------------------+
        |                    |                   |
+-------v------+   +---------v--------+   +------v------+
|              |   |                  |   |             |
| *NIX Kernel  |   | Remote dataplane |   | ........... |
|              |   |                  |   |             |
+--------------+   +------------------+   +-------------+
```

This multi-process architecture provides:

- **High resiliency** â€” an error, crash, or exploit in one protocol daemon generally does not affect the others.
- **Modularity** â€” new protocols can be implemented as new daemons and tied into the suite without modifying existing code.
- **Extensibility** â€” the design makes it straightforward to add new protocol daemons to FRR.

## Zebra: the routing manager

Zebra is the core daemon of FRR. It acts as a broker between the individual protocol daemons and the OS kernel's routing table. Protocol daemons submit routes to Zebra, which then installs them into the kernel. Zebra also handles:

- Interface management (flags, addresses)
- Static route configuration
- Redistribution of routes between protocols
- Kernel Netlink socket communication on Linux (via `CONFIG_NETLINK` and `CONFIG_RTNETLINK`)
- VRF (Virtual Routing and Forwarding) support on Linux kernels â‰Ą 4.8
- MPLS forwarding on Linux kernels â‰Ą 4.5

<Tip>
  You almost always need Zebra running. It can be disabled (`--disable-zebra`) only in specialized scenarios such as running `bgpd` as a standalone BGP route server.
</Tip>

## vtysh: the unified CLI

All FRR daemons can be managed through a single integrated shell called **vtysh**. Rather than requiring you to connect to each daemon's individual VTY port, vtysh connects to all running daemons via UNIX domain sockets and proxies your commands to the appropriate daemon.

vtysh also supports an **integrated configuration mode**, allowing you to manage all daemons from a single `frr.conf` file instead of maintaining a separate config file for each daemon (`zebra.conf`, `bgpd.conf`, etc.).

```bash
# Launch the unified shell
vtysh

# Inside vtysh â€” commands are routed to the correct daemon automatically
Router# show ip bgp summary
Router# show ip ospf neighbor
Router# configure terminal
Router(config)# router bgp 65001
```

## Supported platforms

FRR supports the following operating systems:

| Platform | Status |
|----------|--------|
| GNU/Linux | Fully supported; primary development platform |
| FreeBSD | Fully supported |
| NetBSD | Fully supported |
| OpenBSD | Fully supported |
| Solaris | Best-effort support |
| MacOS | Best-effort support |

Tested compilers include **GCC**, **Clang (LLVM)**, and **Intel ICC**.

<Info>
  Platform-dependent code is largely limited to the Zebra daemon. Protocol daemons are mostly platform-independent, making it relatively straightforward to port FRR to new platforms.
</Info>

## Two user modes

FRR follows a Cisco-like privilege model with two VTY access levels:

- **Normal (view) mode** â€” read-only access for viewing routing state and system status.
- **Enable mode** â€” read-write access for making configuration changes.

This is independent of UNIX user accounts, giving network administrators controlled access to the router without requiring root shell access.
